WHAT THIS PROJECT IS
--------------------
Cardistry was a 2D CSS playing-card toy. It is being rebuilt into a FULL
WebGL app (react-three-fiber) that is BOTH:
  (1) a 3D card visualizer (spread the deck in different arrangements), and
  (2) an interactive teacher for how to shuffle (riffle, overhand, wash,
      hindu, faro...) as guided, scrubbable, step-by-step 3D lessons.

Aesthetic: dark casino "oxblood felt under a gold spotlight." Bespoke CSS
(NO Tailwind/shadcn). Fonts: Fraunces (display), Inter (body), JetBrains
Mono (data/labels). Single accent color = gold. Suit glyphs (spade/heart/
diamond/club) are the structural motif (difficulty tiers, brand).

Stack: React 19 + Vite. three@0.183, @react-three/fiber@9, @react-three/
drei@10, @react-three/rapier@2 (rapier still UNUSED — reserved for Phase 6).
zustand@5 for state. Deploys to Vercel (@vercel/analytics wired in App.jsx).

Full design doc / plan: /Users/shar/.claude/plans/glowing-puzzling-sky.md
(Read it — it explains the architecture rationale in depth.)

THE CORE ARCHITECTURAL RULE (do not violate):
  - Discrete/slow state (mode, deck order, camera preset, player step/speed)
    -> zustand. Per-frame state (each card's position/rotation/bend) ->
    imperative refs written through cardRegistry. NEVER call setState inside
    useFrame (the player mirrors its time cursor to the store at ~12Hz only).
  - Drivers (VisualizerDriver, LessonRunner, future PlaygroundDriver) each own
    the cards via their own useFrame; only one is mounted at a time
    (SceneController switches by store.mode). Cards never remount.
  - All card/hand rotation uses QUATERNIONS (slerp), not Euler, to avoid
    gimbal flips.

HOW TO ADD A NEW LESSON (schema):
  A lesson = { id, title, technique, difficulty:'beginner'|'intermediate'|
  'advanced', randomizes:string, seed:number, cameraPreset, summary, facts:
  [string], build(deck, ctx) -> steps[] }. ctx.rng is a seeded PRNG (use it
  for ALL randomness so compilation stays deterministic).
  Each step is one of:
    { kind:'move', id, label, duration, ease, to:(deck)=>poses | poses,
      bend?, midBend?, reorder?:(deck)=>newDeck, camera?, annotations? }
    { kind:'riffle', id, label, duration, ease, midBend?, arcLift?,
      camera?, annotations? }   // staggered interleave cascade (uses riffleOrder)
    { kind:'hold', id, label, duration, annotations? }
  A pose = { id, pos:THREE.Vector3, quat:THREE.Quaternion, bend:number }.
  annotations: [{ text, at:[x,y,z], appearAt:0..1, until:0..1 }].
  Register it in lessons/catalog/index.js LESSONS array.

HOW THE BEND SHADER WORKS:
  uBend is signed curvature (1/R). The card (long axis = local Y) bows into an
  arc; normals are rotated so lighting stays correct. Bigger |uBend| = more
  bow. Drivers set it via setMaterialBend(material, bend) or the sampled pose's
  .bend. sampleTrack adds a sin() mid-flight curl (midBend) and vertical arc
  (arcLift) during a riffle segment.

CONTENT ACCURACY (already researched — keep these correct):
  - ~7 riffle shuffles randomize 52 cards (Bayer–Diaconis cutoff).
  - Overhand/Hindu only transport blocks -> very weak (~thousands needed).
  - Faro is deterministic/reversible; 8 perfect out-faros restore the deck.
  - Card wash is one of the strongest physical randomizers.
  - Card spring: elastic bend stores energy; past the yield point it creases
    (permanent) and ruins the card.

HOW TO RUN + VERIFY
========================================================================
  npm install        (deps already in package.json incl. zustand)
  npm run dev        -> http://localhost:5173
  npm run lint       (keep clean)
  npm run build      (keep clean)

Engine verification (recommended — this is the crux to protect):
  For any new lesson, assert with the compiled Track:
    - determinism: compile twice, finalDeck order identical
    - deck integrity: hasSameCardSet(createDeck(), track.finalDeck) && len 52
    - scrub fidelity: sample at time T, jump to end, sample T again -> identical
      poses (zero drift) — this is what makes the scrubber trustworthy
    - no NaN poses across the timeline
  (These can be run with `npx vite-node <script.mjs>` since the engine modules
  are pure and import three for math.)

NOTE: the environment used so far had NO browser automation, so the 3D was
verified by logic/build/math only — view it in a real browser and art-direct
the felt/lighting/cards/hands there.
